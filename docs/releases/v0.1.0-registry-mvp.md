# v0.1.0-registry-mvp Release Notes

发布日期：2026-06-23

## 1. 版本定位

`v0.1.0-registry-mvp` 是 ToolVault 的第一个 Registry MVP 版本。

本版本只交付 Tool Registry 的内部基础能力，用于建立工具元数据、生命周期状态、查找和内存实现的最小可验证基线。它仍然是仓库内部模块，不是公开 API、网络服务或完整 Tool Control Plane。

ToolVault 的长期定位仍是 AI Agents 的 Tool Control Plane；但本版本只覆盖 v1 core 中的 Tool Registry slice。

## 2. 本版本完成了什么

- 定义了 `ToolSpec` 模型及校验规则，位置在 `internal/registry/spec`。
- 定义了 Registry 接口、请求类型、选项类型、生命周期辅助函数和错误契约，位置在 `internal/registry`。
- 实现了标准库 in-memory Registry，位置在 `internal/registry/memory`。
- 支持 Registry 负责的生命周期状态：`draft`、`active`、`deprecated`、`disabled`。
- 支持 `Register`、`Get`、`List`、`Update`、`Delete`、`SetStatus`。
- 支持 opaque version 校验，并在 `Update` 中通过 `ExpectedVersion` 做乐观更新检查。
- 支持 M1 hard delete 语义。
- 支持稳定 list 排序、status/tag 过滤、重复 ID/name 检查，以及对 slice/map 字段的 defensive copy。
- 补充了 Registry 示例、验收清单和 review 记录。

## 3. 本版本没有做什么

本版本没有实现以下能力：

- Gateway 请求入口或受控工具调用。
- Runtime 执行、运行时选择或执行生命周期治理。
- Policy 评估、鉴权或授权逻辑。
- Credential 注入、密钥读取或密钥存储。
- MCP、OpenAI Function Calling 或其他协议适配器。
- 数据库存储或持久化 Registry。
- Observability pipeline、审计事件、指标或 tracing。
- Dashboard、Sandbox、Streaming、Tool Composition、Marketplace 或 Web UI。
- Agent Framework、LLM Gateway、Workflow Engine、Vector DB 或 Prompt Management Platform 能力。

## 4. Registry 当前能力

当前 Registry 能保存和管理内部 `ToolSpec`：

- 必填字段校验：`ID`、`Name`、`Version`、`Status`。
- machine-readable 名称校验：`Name` 和 action name 必须以小写字母开头，只包含小写字母、数字、下划线或连字符。
- opaque version 校验：版本不能为空，且不能包含空白或控制字符。
- action 校验：至少一个 action，action name 不重复，action description 必填。
- metadata 校验：最多 32 项，key 最长 64 bytes，key 不能为空且不能带首尾空白。
- 注册状态限制：初始状态只允许 `draft` 或 `active`。
- 生命周期转换：
  - `draft` -> `active` 或 `disabled`
  - `active` -> `deprecated` 或 `disabled`
  - `deprecated` -> `disabled`
  - 相同状态重复设置允许
- 查询能力：
  - 按 ID 获取单个 tool。
  - 列出全部非删除 tool。
  - 按 status 和 tag 过滤 list 结果。
  - list 结果按 ID 稳定排序。
- 更新能力：
  - `Update` 只更新 M1 可变元数据字段。
  - `ID`、`Name`、`Status` 不通过 `Update` 修改。
  - 调用方必须提供 `UpdateOptions.ExpectedVersion`。
  - 成功更新必须提供新的 opaque `ToolSpecUpdate.Version`。
- 错误契约：
  - `already_exists`
  - `not_found`
  - `invalid_spec`
  - `invalid_state_transition`
  - `version_conflict`

## 5. 使用方式入口

当前入口是仓库内部 Go 包：

- `internal/registry/spec`：`ToolSpec`、`ToolStatus`、`ToolAction` 和 validation。
- `internal/registry`：Registry interface、request/option 类型、生命周期规则和错误类型。
- `internal/registry/memory`：标准库 in-memory implementation。

最小示例见：

- `docs/examples/registry-basic.md`
- `internal/registry/memory/example_test.go`

示例覆盖创建 in-memory Registry、注册 tool、按 ID 读取、按 tag list，以及将 `draft` 状态切换到 `active`。

## 6. 已知限制

- Registry 仍是 internal module，不是 public package。
- in-memory implementation 不提供持久化；进程结束后数据丢失。
- `Delete` 是 M1 hard delete，没有 soft delete、restore 或 tombstone。
- 当前没有网络 API、CLI 或 Gateway 接入层。
- 当前没有 Runtime、Policy、Credential、Protocol Adapter 或 Observability 模块消费 Registry。
- `ToolSpec` contract 相比 Proposal 001 有扩展：包含 `Actions` 和 `Metadata`。
- `Update` 引入了 `ExpectedVersion` / `VersionConflict` 乐观更新语义。
- 生命周期变更通过 `SetStatus` 完成，而不是通过 `Update`。
- in-memory Registry 除了拒绝重复 `ID`，也拒绝重复 `Name`。
- 上述 contract deviations 已在 M1 acceptance 中记录；M2 使用 Registry 前需要人工确认接受或收窄。

## 7. 下一阶段方向

下一阶段建议进入 M2 Gateway planning，而不是直接扩展 Registry 行为。

M2 前应先完成：

- 人工确认或收窄 M1 acceptance 记录的 Registry contract deviations。
- 明确 Gateway 如何消费 Registry，以及模块依赖方向。
- 明确 Registry 仍保持内部模块，除非有批准的边界变更 Proposal。
- 继续避免引入 Runtime、Policy、Credential、Protocol Adapter、Observability 或 Web UI 的实现，直到对应 milestone 和 proposal 获批。
